# YIP Platform — Phase 2 Specification

> Generated by SDD workflow on 2026-03-31

## Overview

Phase 2 extends the YIP Platform with 6 features that complete the mock parliament simulation: Question Hour management, Bill drafting workspace, digital voting on phones, WhatsApp notifications, multi-level pipeline (Chapter → Regional → National), and certificate generation.

---

## Feature Prioritization (by event-day impact)

| Priority | Feature | Why This Order |
|----------|---------|----------------|
| P1 | **Question Hour** | Used in every event (Day 2 morning). Currently manual — biggest pain point left |
| P2 | **Digital Voting** | Used for Speaker Election (Day 1) + Bill Voting (Day 2). Replaces hand-counting |
| P3 | **Bill Drafting** | Used Day 2 afternoon. Students need a workspace to draft bills collaboratively |
| P4 | **Certificate Generation** | Used after every event. High-visibility output for students and schools |
| P5 | **Multi-Level Pipeline** | Used between events. Connects Chapter → Regional → National progression |
| P6 | **WhatsApp Notifications** | Nice-to-have. Access codes, schedule, results sent via WhatsApp |

## Functional Specification

### F1: Question Hour Digital Management

**What it replaces:** Currently, questions are collected via a Google Form, manually filtered by organizers into starred/unstarred, and managed on paper during the live session.

**How it works:**
1. **Pre-event (3-5 days before):** Students submit questions via the platform (using their access code)
2. **Organizer filters:** Reviews submitted questions, marks as starred (important) or unstarred, assigns queue order
3. **Live event (Day 2):** Question Hour agenda item activates → organizer sees question queue → advances through questions → minister responds → tracked

**Database:** New `questions` table (already referenced in spec but not yet created)

#### Screens:

**S15: Student Question Submission** (`/me/questions`)
- Shows when event is in registration_open/registration_closed status
- Form: Question text (textarea), Directed to Ministry (dropdown of 8 ministries)
- Student can submit up to 3 questions
- Shows their submitted questions with status

**S16: Organizer Question Management** (`/dashboard/events/[id]/questions`)
- New tab in event nav
- Table of all submitted questions: Student Name, Party, Constituency, Ministry, Question text, Type (starred/unstarred), Status, Queue Order
- Bulk actions: Mark as starred, Mark as unstarred, Approve, Reject
- Drag-and-drop or number input to set queue order
- Filter by ministry, party, status

**S17: Live Question Hour** (integrated into Control Panel)
- When current agenda item type = 'question_hour':
  - Control panel shows question queue instead of speaker queue
  - Current question displayed: student name, party, constituency, question text, directed to ministry
  - "Minister Responds" timer (configurable, default 2 min)
  - "Next Question" / "Skip Question" buttons
  - Mark as "Answered" with optional answer summary
- Projector view shows: current question, student name, ministry name

### F2: Digital Voting (Speaker Election + Bill Voting)

**What it replaces:** Hand-raising or paper ballot counting

**How it works:**
- When organizer opens a vote → all participants see a vote button on their phone
- Each participant casts one vote → results tallied in real-time → organizer reveals results

**Existing infrastructure:** `votes` table already exists with `vote_type` supporting 'speaker_election' and 'bill_vote'

#### Screens:

**S18: Vote Casting** (`/me` — contextual, appears when vote is open)
- When event has an active vote, participant's page shows a voting card
- **Speaker Election:** Shows list of 6 speaker candidates → tap to select one → confirm → vote cast
- **Bill Vote:** Shows bill title + summary → "Aye" / "Nay" / "Abstain" buttons → confirm → vote cast
- After voting: "Your vote has been recorded" ✓
- Cannot change vote after casting

**S19: Vote Management** (Control Panel integration)
- "Open Vote" button in control panel (appears during speaker_election or bill_presentation agenda items)
- Shows live vote count (hidden from participants until reveal)
- "Close Voting" → stops accepting votes
- "Reveal Results" → shows on projector + control panel
- Results: candidate names with vote counts, or Aye/Nay/Abstain counts with pass/fail

**S20: Projector Vote Display**
- When vote is open: "VOTING IS OPEN — Cast your vote on your phone"
- After reveal: animated results display (bar chart or count-up animation)

### F3: Bill Drafting Workspace

**What it replaces:** Paper templates and separate documents

**How it works:**
- After committee discussions (Day 1), bill drafting committee (4 MPs per side) collaborates on a bill
- Each party submits one bill using the official template
- Bills are presented on Day 2 afternoon

**Database:** New `bills` table (already referenced in schema but not yet created — wait, check... yes, it's NOT in the migration. Need to create.)

#### Screens:

**S21: Bill Drafting** (`/me/bill`)
- Only visible to participants with role = 'bill_committee'
- Template form matching YIP bill format:
  - Party Name (auto-filled)
  - Bill Title
  - Objective (textarea)
  - Problem Statement (1-2 lines)
  - Key Provisions (3 inputs)
  - Expected Impact (textarea)
  - Implementation Mechanism (textarea)
  - Lead Drafter, Presenter 1, Presenter 2, Policy Researcher (auto-filled from committee members)
- Auto-save as draft
- "Submit Bill" when ready
- Both ruling and opposition committees draft independently

**S22: Bill Review** (`/dashboard/events/[id]/bills`)
- New tab in event nav
- Shows submitted bills from both sides
- Organizer can approve/reject/request revisions
- Status flow: drafting → submitted → approved → presented → voted

**S23: Bill Presentation** (Control Panel integration)
- During bill_presentation agenda item:
  - Show bill details on projector
  - Presenter 1 speaks (use speaker queue)
  - Opposition challenges
  - Presenter 2 responds
  - Open floor debate
  - Vote (integrates with F2 voting)

### F4: Certificate Generation

**What it replaces:** Manual certificate creation in Canva/Word

**How it works:**
- After results are published, organizer generates certificates
- Two types: Participation Certificate (all students) and Award Certificate (winners)
- PDF download, individually or as batch ZIP

#### Screens:

**S24: Certificate Generator** (`/dashboard/events/[id]/certificates`)
- New tab in event nav (visible after results published)
- Preview certificate template with event details
- "Generate All Certificates" → creates PDFs
- Download individual or batch ZIP
- Certificate content:
  - **Participation:** "This is to certify that [Name] of [School] participated as [Role] representing [Constituency] in the Young Indians Parliament [Chapter/Regional/National] Round held at [Venue] on [Dates]"
  - **Award:** Same + "and was awarded [Award Category] for outstanding performance"
- Signed by: Chapter Chair (customizable name field)

**Implementation:** Use @react-pdf/renderer or html2canvas + jsPDF for client-side PDF generation. No server-side PDF needed.

### F5: Multi-Level Pipeline (Chapter → Regional → National)

**What it replaces:** Manual tracking of winners across levels in spreadsheets

**How it works:**
- After chapter event, organizer marks top performers as "qualified for regional"
- National admin creates regional events
- Qualified students are auto-imported into regional events with their chapter performance data
- Same flow for Regional → National

**Existing infrastructure:** `participants.qualified_for_next` field already exists. `events` table has `level` and `season_id`.

#### Screens:

**S25: Qualification Selection** (`/dashboard/events/[id]/results` — extended)
- After publishing results, new section: "Select Qualifiers for Regional Round"
- Checkboxes on leaderboard to mark qualifiers
- "Promote X Students to Regional" button
- Shows which students were already promoted

**S26: National Admin Dashboard** (`/dashboard/admin`)
- New role: national_admin (add to auth)
- See all events across all chapters for the season
- Create regional events
- Import qualified students from chapter events
- Create national event from regional qualifiers
- Season-wide statistics: total participants, chapters completed, etc.

**S27: Pipeline View** (`/dashboard/admin/pipeline`)
- Visual pipeline: Chapter Events → Regional Events → National Event
- Show student count at each stage
- Track which chapters have completed, which are pending
- Click through to any event

### F6: WhatsApp Notifications

**What it replaces:** Manual WhatsApp messages from organizer's phone

**How it works:**
- Organizer sends bulk messages to participants via the platform
- Templates: access code delivery, schedule reminder, results notification
- Uses WhatsApp Business API (or the user's existing WhatsApp MCP integration)

#### Screens:

**S28: Notification Center** (`/dashboard/events/[id]/notifications`)
- New tab in event nav
- Pre-built templates:
  - "Your Access Code" — sends code + event details
  - "Event Reminder" — sends schedule for tomorrow
  - "Results Available" — sends score + rank
- "Send to All" or "Send to Selected" (filter by party/role/committee)
- Message preview before sending
- Delivery status tracking

[ASSUMPTION]: WhatsApp integration will use the existing MCP WhatsApp tools available in the user's Claude Code setup, or a simple WhatsApp Business API integration. Exact API TBD.

## Scope Boundaries

**In scope (Phase 2):**
- Question Hour: submission, filtering, live queue, minister response tracking
- Digital Voting: speaker election + bill voting on phones, real-time tallying
- Bill Drafting: template form, collaborative drafting, submission, presentation flow
- Certificates: participation + award PDFs, batch download
- Pipeline: chapter→regional→national, qualifier selection, auto-import
- WhatsApp: bulk notifications with templates

**Explicitly out of scope (Phase 2):**
- Offline mode (IndexedDB)
- Payment/registration fee collection
- Analytics dashboard beyond pipeline view
- Student preparation materials/study hub
- Organizer checklist tracker (47-item)
- AI-assisted debate coaching or speech feedback
- Mobile app (stays web-only)

## Technical Plan

### New Database Tables

```sql
-- Questions for Question Hour
CREATE TABLE questions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  submitted_by UUID NOT NULL REFERENCES participants(id) ON DELETE CASCADE,
  directed_to_ministry ministry_type NOT NULL,
  question_text TEXT NOT NULL,
  question_type TEXT CHECK (question_type IN ('starred', 'unstarred')),
  status TEXT DEFAULT 'submitted' CHECK (status IN ('submitted', 'approved', 'rejected', 'queued', 'asked', 'answered')),
  queue_order INTEGER,
  answer_summary TEXT,
  filtered_by UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- Bills for Bill Drafting
CREATE TABLE bills (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  party_side party_side NOT NULL,
  title TEXT NOT NULL,
  objective TEXT,
  problem_statement TEXT,
  provisions JSONB DEFAULT '[]'::jsonb,
  expected_impact TEXT,
  implementation TEXT,
  lead_drafter UUID REFERENCES participants(id),
  presenter_1 UUID REFERENCES participants(id),
  presenter_2 UUID REFERENCES participants(id),
  policy_researcher UUID REFERENCES participants(id),
  status TEXT DEFAULT 'drafting' CHECK (status IN ('drafting', 'submitted', 'approved', 'presented', 'voting', 'passed', 'rejected')),
  votes_for INTEGER DEFAULT 0,
  votes_against INTEGER DEFAULT 0,
  votes_abstain INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- Active vote sessions (tracks when voting is open)
CREATE TABLE vote_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  agenda_item_id UUID NOT NULL REFERENCES agenda_items(id),
  vote_type TEXT NOT NULL CHECK (vote_type IN ('speaker_election', 'bill_vote')),
  status TEXT DEFAULT 'open' CHECK (status IN ('open', 'closed', 'revealed')),
  -- For speaker election: candidates stored in config
  -- For bill vote: bill_id reference
  bill_id UUID REFERENCES bills(id),
  config JSONB DEFAULT '{}'::jsonb,
  opened_at TIMESTAMPTZ DEFAULT now(),
  closed_at TIMESTAMPTZ,
  revealed_at TIMESTAMPTZ
);

-- Notification log
CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  template TEXT NOT NULL,
  sent_to_count INTEGER DEFAULT 0,
  sent_at TIMESTAMPTZ DEFAULT now(),
  status TEXT DEFAULT 'sent',
  config JSONB DEFAULT '{}'::jsonb
);
```
