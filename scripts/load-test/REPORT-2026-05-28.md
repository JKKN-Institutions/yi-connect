## Verdict: GREEN

The deployed YIP stack handles the targeted ~10× chapter-day load on the
public, anonymous pages with margin to spare:

- Sustained 400–800 req/sec on edge-cached pages.
- p99 < 1.1 s on the heaviest scenario; < 500 ms on three of four.
- No `5xx` of any kind. No timeouts. No queueing artefacts.

The single most concerning observation: **the landing page Max latency hit
2 144 ms** on one outlier request during the 100-connection burst. That is
likely a Vercel cold-start or a single dropped TCP retransmit — a one-off
across 11 689 requests is acceptable, but worth re-checking at the same
hour on June 4.

## What this test did NOT cover

This was anonymous edge-cache load. The June 4–5 risk is not the edge — it
is **authenticated workloads**:

1. **150 access-code logins in the first 5 min** of a chapter session.
   Each goes through a server action that touches Supabase. Not tested
   here because it would mutate data.
2. **Concurrent jury scoring writes** during live rounds — 5–7 jurors per
   round × 10 chapters = ~50 simultaneous writes per scoring window.
3. **Supabase Realtime broadcast** for the agenda state machine. The
   organiser-control "advance agenda" event fans out to every connected
   client per event.

These require seeded test data + scripted login flows. **Recommend a
Phase 20 follow-on**: build a seeded dummy event, generate 150 access
codes, write a `k6` (or autocannon-with-headers) script that does the full
login → fetch dashboard → submit-score loop, and run that at 10× before
June 4.
