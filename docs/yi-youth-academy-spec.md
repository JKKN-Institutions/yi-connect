## Overview

Yi Youth Academy is a physical, branded leadership space ("Yi – [Institution Name] Youth Academy") inside a YUVA partner college, where the local Yi chapter runs **cohort-based certificate programs** (seven categories per the national Program Creation Template: Entrepreneurship, Innovation, Learning, Accessibility, Climate Change, Health, Road Safety) for students in the Yi YUVA network. This platform digitizes the full lifecycle: National authors program templates and creates academies (one per chapter, with logo; onboarding SoP/MoU happens offline) → Chapter schedules program runs → Students apply via a public landing page → Chapter/Institution accept applications and form cohorts → Mentors deliver sessions, share materials, mark attendance, review submissions → Students track progress and receive e-certificates → National monitors usage norms and outcomes quarterly. Four programs are ready to load on day one (3 Entrepreneurship, 1 Innovation).

**Operating model: template → instance.** National defines the curriculum (program + session structure + durations + takeaways). Chapters instantiate it as a *run* at a specific academy with concrete dates, venue, and a mentor per session. Cohort-based delivery is mandatory; programs are certificate programs approved by the Yi National Team.

## Functional Specification

### User Roles

| Role | Identity representation | Can Do | Cannot Do |
|------|------------------------|--------|-----------|
| **National admin** (Yi YUVA National Team) | `yi_directory.role_assignments` app=`yuva`, role=`yuva_super_admin` / `yuva_admin` (global scope) | Author/approve/archive program templates with session structures; **create/edit academies (chapter, institution, display name, logo) — creation IS the approval**; activate/deactivate academies; view all academies/runs/cohorts; usage-norm compliance dashboard (3 engagements/month, 30 active days/year); quarterly review metrics + export; chapter-admin role grants happen via the platform directory console (`/admin/directory`), not in-app | Cannot edit another app's data (siloed visibility); cannot bypass the platform-super-admin-only directory editing rules |
| **Chapter admin** (Yi Youth Academy chapter team) | app=`yuva`, role=`chapter_admin`, `yi_chapter` set | View their academy/academies (national-created); assign the optional institution coordinator; build the mentor network — the **Mentor YUVA Network** (invite mentors); create runs from approved templates; schedule sessions (date/time/venue, mentor per session); publish runs to the landing page; review/accept/reject applications; form cohorts; mark attendance; review per-session student work; issue e-certificates; view chapter-level reports | Cannot touch other chapters' academies/runs (fail-closed chapter scope); cannot create/edit program templates; cannot create or edit academies (national-only) |
| **Institution coordinator** (Faculty Coordinator — full co-manager) | app=`yuva`, role=`institution_coordinator`, `yi_chapter` set + bound to one academy | Everything a chapter admin can do **for their own academy only**: schedule sessions, review applications, form cohorts, attendance, per-session student work, certificates, view reports | Cannot create or edit academies (national-only); cannot invite mentors to the chapter network (chapter-owned); cannot see other academies |
| **Mentor** | app=`yuva`, role=`mentor`, `yi_chapter` set. **Mentors can be ANYONE** — no Yi membership required, no national vetting (chapters invite freely) | Edit own profile (bio, photo, expertise — shown publicly on program pages and the public **Mentor YUVA Network** page); see assigned sessions + cohort rosters; upload session materials; mark attendance for own sessions; message the cohort (thread); review per-session student work and give feedback/evaluation | Cannot see runs/cohorts they are not assigned to; cannot accept/reject applications; cannot issue certificates |
| **Student** (Yi YUVA students only) | **Subject, not role-holder**: `yi_directory.people` row via `resolvePerson()` + `yuva.enrollments`; login provisioned on acceptance | Browse public landing page; apply to a published run (no login needed); after acceptance: log in, see schedule/timelines/deadlines, view mentor profiles, download materials, post in cohort thread, upload submissions, track own progress/attendance, download e-certificate | Cannot see other students' submissions/PII; cannot apply twice to the same run; cannot access anything before acceptance |
| **Visitor** (anonymous) | none | View landing page, published program/run details, mentor public profiles; start an application | Everything else |

**Fail-closed rule (project-wide):** every gate denies on null/unknown scope and renders an explicit `Forbidden403` — never a silent redirect.

### Screens / Pages

All routes mount under `/youth-academy`. Personas follow the Yi-Future persona-route pattern (`national/`, `chapter/`, `mentor/`, `me/`). The institution coordinator does NOT get a duplicate page tree: `/youth-academy/institution` is a thin dashboard that deep-links into the `chapter/*` screens, whose gate accepts chapter scope OR coordinator-academy scope (plan decision, reconciled).

#### Public landing page
- **URL:** `/youth-academy`
- **Who sees it:** everyone
- **What it shows:** Yi YUVA / Youth Academy branding, what the Academy is (concept-note positioning), grid of **published runs** (program name, category badge, academy display name, city, start–end dates, total hours, seats indicator, application deadline), filter by category/city; an **"Our Network" section** showcasing ALL active academies (logo, academy name, chapter, institution when known) — the list of academies showcased as our existing network; links to the public **Mentor YUVA Network** page
- **Actions:** open run detail; Apply; open the Mentor YUVA Network page
- **Edge cases:** no published runs → branded empty state with "programs coming soon"; run past application deadline or full → card shows "Applications closed" (not hidden)

#### Run detail (public)
- **URL:** `/youth-academy/programs/[runId]`
- **Who sees it:** everyone (only `published`+ statuses; draft → 404)
- **What it shows:** program summary + takeaways, total hours, session-by-session schedule (name, date, time, duration, venue, mentor name+photo→profile), academy/institution info, application window, cohort announcement date ("results announced by …"), seats
- **Actions:** Apply (disabled with reason if closed/full)
- **Edge cases:** session without an assigned mentor shows "Mentor to be announced"; run unpublished after publish → public page returns 404, applicants keep access to their application status page

#### Mentor YUVA Network (public)
- **URL:** `/youth-academy/mentors`
- **Who sees it:** everyone
- **What it shows:** ALL active mentors with a public profile, across chapters (photo, name, expertise, organization, chapter) — the public showcase of the **Mentor YUVA Network**; linked from the landing page
- **Edge cases:** mentor with `is_public=false` or deactivated → hidden; zero public mentors → branded empty state

#### Application form (public)
- **URL:** `/youth-academy/apply/[runId]`
- **Who sees it:** everyone while the run is accepting applications
- **What it shows:** multi-step form — personal details (name, email, phone, DOB optional), academic (institution dropdown sourced from YUVA institutions + "Other", degree/department, year of study), motivation ("why do you want to join"), Yi YUVA membership claim (member / want to join), declaration/consent checkbox
- **Actions:** submit → confirmation screen + confirmation email with a status-check link
- **Edge cases:** duplicate application (same email+run) → "you've already applied" with status link instead of a second row; run closes between page load and submit → friendly rejection; email service down → application still saved, email queued (durable queue)

#### Application status (public, tokenized)
- **URL:** `/youth-academy/applications/[token]`
- **Who sees it:** the applicant (unguessable token from confirmation email)
- **What it shows:** pending ("decisions announced by {cohort announcement date}") / accepted — login details arrive by email when your cohort is formed / rejected (respectful message)
- **Edge cases:** invalid token → 404, no data leak

#### Login
- **URL:** `/youth-academy/login`
- **Who sees it:** everyone
- **What it shows:** two paths — **Students:** access code (emailed when your cohort is formed) or email OTP; **Staff (national/chapter/institution/mentor):** Google OAuth / email — the existing shared Supabase auth flows
- **Edge cases:** student with no acceptance → clear error; deactivated person → fail closed with explicit message; logged-in user landing here → redirect to their persona dashboard

#### National — dashboard
- **URL:** `/youth-academy/national`
- **Who sees it:** `yuva_super_admin`, `yuva_admin`
- **What it shows:** KPI cards (active academies, runs in progress, sessions conducted this month, students engaged, students certified); **usage-norm compliance table** per academy (engagements this month vs ≥3 — one each E/I/L; active days YTD vs ≥30) with red/amber/green; quarterly review summary + CSV export
- **Edge cases:** zero academies → "create your first academy" empty state (national creates academies); norms computed from completed sessions only

#### National — program templates
- **URL:** `/youth-academy/national/programs` (+ `/new`, `/[id]`)
- **What it shows:** template list (title, category, sessions count, total hours auto-computed, status draft/approved/archived, runs count); editor: title, category (the seven categories from the national Program Creation Template: Entrepreneurship / Innovation / Learning / Accessibility / Climate Change / Health / Road Safety), **Program Objective**, summary (UI hint: 100–150 words), takeaways list, **session structure builder** (ordered sessions: name, duration in minutes, learning objective, description, **session document upload** — course material per session, uploaded by national to the private `yuva-materials` bucket under a `program/` prefix — and an "expects student work" toggle; 8–10 sessions typical) with live total-hours readout (total duration stays auto-calculated from sessions — the template allows it)
- **Actions:** create, edit, approve (makes it instantiable by chapters), archive (blocks NEW runs, existing runs unaffected)
- **Edge cases:** editing an approved template that has live runs → structure changes apply to NEW runs only (runs snapshot their session structure at creation); cannot approve a template with zero sessions

#### National — academies (national-created)
- **URL:** `/youth-academy/national/academies` (+ `/new`, `/[id]`)
- **What it shows:** create/edit academy — pick chapter from the Yi chapter list, optional institution (institution + display name editable later), display name (default "Yi [Chapter] Youth Academy" until an institution is attached, then "Yi – [Institution] Youth Academy"), **logo upload** (per-academy combo logo: youth-network + partner-institution logo, uploaded by national to the public `yuva-public` bucket); academy list with per-academy compliance summary; detail shows runs and qualitative outcomes (free-text qualitative outcomes per academy, editable by chapter/national, included in the quarterly CSV)
- **Actions:** create academy (**national creating the academy IS the approval** — there is no separate approval step), edit, upload/replace logo, activate/deactivate
- **Edge cases:** institution already attached to another academy → block duplicate; deactivating an academy with a live run → blocked until run completes; academy changes are audit-logged via `yuva.audit_log`

#### Chapter — dashboard
- **URL:** `/youth-academy/chapter`
- **Who sees it:** `chapter_admin` (their `yi_chapter` only)
- **What it shows:** their academy/academies (national-created), active runs, pending-applications count, upcoming sessions this week, norm compliance for their academies
- **Edge cases:** user with role but null chapter scope → DENY (fail closed); chapter with no academy yet → "your academy is set up by the Yi National team" empty state (chapters cannot create academies)

#### Chapter — academy view (read-only)
- **URL:** `/youth-academy/chapter/academies` (+ `/[id]`)
- **What it shows:** read-only view of their academy/academies (display name, logo, institution, capacity norm, active status) + qualitative outcomes (editable); assign the institution coordinator (OPTIONAL — name + email → creates their login + role; an academy can operate with no coordinator at all, the chapter runs the show through its own logins; Director 2026-06-10)
- **Actions:** assign/replace coordinator; edit qualitative outcomes — academy create/edit/logo/activation are national-only
- **Edge cases:** coordinator email already in directory → link to existing person instead of duplicate; chapter attempting an academy create/edit → Forbidden403 (national-only gate)

#### Chapter — Mentor YUVA Network
- **URL:** `/youth-academy/chapter/mentors`
- **What it shows:** the chapter's **Mentor YUVA Network** roster (name, expertise, sessions delivered, upcoming sessions); mentors can be ANYONE — no Yi membership, no national vetting; active public profiles appear on the public `/youth-academy/mentors` page
- **Actions:** invite mentor (name + email → person + `mentor` role + invite email), deactivate mentor
- **Edge cases:** deactivating a mentor with future assigned sessions → warning listing sessions needing reassignment; invite to an email that's already a mentor in another chapter → allowed (multi-chapter mentor, separate role rows)

#### Chapter / Institution — runs (program scheduling)
- **URL:** `/youth-academy/chapter/runs` (+ `/new`, `/[id]`) — institution coordinator sees the same screens scoped to their academy
- **What it shows:** create run = pick approved template + academy → run is created in `draft` with sessions copied from the template; scheduling table: per session set facilitator/mentor (from chapter network), exact date+time, venue (defaults to academy), and **Additional Remarks** (chapter-filled, per session); run settings: application window (open/close dates), **cohort announcement date** (the date applicants are told decisions arrive and codes go out — shown publicly; required to publish; Director 2026-06-10), capacity (default 50 per space norms — soft cap; this is the template Part B "Expected Participants" field), **start/end dates entered by the chapter** (template Part B "Filled by Chapter"); sessions scheduled outside the entered range trigger a warning, not a block
- **Actions:** save schedule, **publish** (validation: every session has date+time, application window set, cohort announcement date set; unassigned mentors allowed → "to be announced"), close applications, unpublish (with confirmation; existing applicants keep status access)
- **Edge cases:** overlapping sessions same academy/venue → warning, not block; editing schedule after publish → allowed, students notified by email; mentor double-booked at same datetime → warning

#### Chapter / Institution — applications review
- **URL:** `/youth-academy/chapter/runs/[id]/applications`
- **What it shows:** queue with applicant details (form answers, institution, YUVA membership claim), filters (pending/accepted/rejected), capacity meter
- **Actions:** accept / reject (optional note), bulk accept/reject; **Form cohort** (locks acceptance set, provisions student logins + access codes, sends acceptance emails, creates the cohort thread)
- **Edge cases:** accept beyond capacity → warning (soft cap); reject-then-change-mind before cohort formation → allowed; after cohort formation, late acceptance = explicit "add to cohort" action; applicant email matching an existing person → `resolvePerson()` match, no duplicate identity

#### Chapter / Institution — cohort management
- **URL:** `/youth-academy/chapter/runs/[id]/cohort`
- **What it shows:** roster (name, institution, attendance %, submissions status, progress %), per-session attendance grid, per-session submissions overview, certificate status
- **Actions:** mark/edit attendance per session (also available to the session's mentor); per-student "resend access code" action; review per-session student work; **complete run**; **issue certificates** (eligibility list with attendance %, default suggestion ≥75% attendance — chapter can override per student; generates PDFs, emails students, increments national "certified" metric)
- **Edge cases:** student drops out → mark `dropped` (kept for history, excluded from progress denominators); certificate re-issue (name spelling fix) → regenerate, same certificate number, audit-logged; issuing before run completion → blocked

#### Mentor — dashboard / sessions / cohort
- **URL:** `/youth-academy/mentor` (+ `/sessions/[id]`, `/cohorts/[runId]`, `/profile`)
- **Who sees it:** `mentor` role; sessions/cohorts they are assigned to only
- **What it shows:** upcoming sessions; session detail (roster, attendance marking, materials upload/list); cohort view (roster shows attendance %, submissions status, and progress % per student — reuses `lib/yuva/progress.ts` — plus message thread and submissions queue **grouped by session** with review/feedback per submission; full run schedule + program summary/takeaways visible read-only — the syllabus view); profile editor (photo, bio, expertise tags, organization — public on program pages)
- **Edge cases:** mentor opens a session they're not assigned to → Forbidden403; mentor uploads material before run published → allowed, visible to students only after publish; submission review after student resubmits → newest version flagged

#### Student — portal
- **URL:** `/youth-academy/me` (+ `/program/[runId]`, `/work`, `/messages`, `/certificate`)
- **Who sees it:** enrolled students (session from acceptance)
- **What it shows:** my program(s): next session card, full schedule/timeline; **per session: name, learning objective, session document download (signed URL), date/time/venue, facilitator** (mentor profile), plus mentor-uploaded materials; **My Work**: one submission slot per session that expects work (upload file or text, see status draft/submitted/reviewed + mentor feedback); messages: cohort thread (read + post); progress: attendance %, per-session submissions completed, overall progress bar; certificate: download when issued
- **Edge cases:** student in multiple runs → program switcher; submission after the session's scheduled date + grace window (7 days default) → allowed, flagged "late" (mentor sees); certificate before issuance → "in progress" state; rejected/never-accepted person hitting `/me` → fail closed to login

### Data Model

All academy tables live in a new **`yuva`** Postgres schema. Roles live in `yi_directory.role_assignments` (NO yuva auth tables). Students are `yi_directory.people` subjects via `resolvePerson()`.

| Entity | Key Fields | Relationships |
|--------|-----------|---------------|
| `yuva.academies` | institution_id (→ `yi.institutions`, NULLABLE — launch academies may have no partner institution yet; institution + display name editable later), chapter (text, yi_chapter value), display_name (default "Yi [Chapter] Youth Academy" until an institution is attached, then "Yi – [Institution] Youth Academy"), is_active (boolean, default true), logo_storage_path (per-academy combo logo, national-uploaded, public bucket), coordinator_person_id, capacity_norm | belongs to institution (optional); has many runs; changes audit-logged via `yuva.audit_log` |
| `yuva.programs` | title, category (entrepreneurship/innovation/learning/accessibility/climate_change/health/road_safety), objective, summary (100–150 words), takeaways (jsonb[]), status (draft/approved/archived), total_minutes (computed), created_by | has many program_sessions, runs |
| `yuva.program_sessions` | program_id, seq, name, duration_minutes, learning_objective, description, document_storage_path (session document uploaded by NATIONAL at template level — course material for that session), expects_submission (bool — national marks which sessions expect student work) | template session structure |
| `yuva.runs` | program_id, academy_id, chapter, status (draft → published → applications_closed → in_progress → completed → certified; cancelled), apply_open_at, apply_close_at, cohort_announce_date, capacity (= template Part B "Expected Participants"), start_date/end_date (ENTERED BY CHAPTER — template Part B), published_at | belongs to program + academy; has many run_sessions, applications, enrollments, threads |
| `yuva.run_sessions` | run_id, seq, name, duration_minutes, learning_objective, document_storage_path, expects_submission (all snapshotted from template), scheduled_at, venue, mentor_person_id (nullable), remarks (chapter-filled "Additional Remarks"), status (scheduled/completed/cancelled) | has many attendance, materials, submissions |
| `yuva.applications` | run_id, person_id (resolved), email, phone, full_name, institution_id/institution_other, degree, year_of_study, motivation, yuva_member_claim, status (pending/accepted/rejected/withdrawn), status_token (unguessable), reviewed_by/reviewed_at, consent_at | unique (run_id, lower(email)) |
| `yuva.enrollments` | run_id, person_id, application_id, status (active/completed/dropped), access_code, certificate_id (nullable) | the cohort; has many attendance, submissions |
| `yuva.attendance` | run_session_id, enrollment_id, present (bool), marked_by, marked_at | unique (run_session_id, enrollment_id) |
| `yuva.materials` | run_session_id, title, storage_path, uploaded_by, visible (bool) | session learning materials |
| `yuva.submissions` | run_session_id (→ run_sessions — per-session student work, no separate assignments system), enrollment_id, version, file_storage_path / text_body, status (draft/submitted/reviewed), submitted_at, is_late (submitted after the session's scheduled date + grace window, 7 days default), feedback, reviewed_by/reviewed_at | unique (run_session_id, enrollment_id, version) |
| `yuva.threads` / `yuva.messages` | thread per run (cohort thread); messages: thread_id, sender_person_id, sender_kind (mentor/student/chapter/institution/national), body, created_at | modeled on `future.message_threads` |
| `yuva.certificates` | enrollment_id, certificate_no (YYA-{year}-{seq}), issued_by, issued_at, pdf_storage_path, revoked (bool) | one per enrollment |
| `yuva.mentor_profiles` | person_id (unique), bio, expertise (text[]), organization, photo_storage_path, is_public | app-specific profile over yi_directory identity |
| `yuva.notification_log` | recipient, type, payload, status (pending/sent/failed), attempts | durable email queue (clone of yi-future pattern) |
| `yuva.audit_log` | actor_person_id, action, entity, entity_id, meta jsonb | per-mutation audit (yip pattern) |

**Metrics derivations (national):** sessions conducted = count of `run_sessions.status='completed'`; engagements/month per academy = completed sessions grouped by month × category — **sessions ARE the engagements; one-off events do not exist for this program**, so the usage norm (3 engagements/month, one each Entrepreneurship/Innovation/Learning) is computed entirely from completed sessions; active days = distinct dates with a completed session; students engaged = distinct enrolled persons; certified = certificates issued.

### Scope Boundaries

**In scope (v1):** everything above — full LMS depth, institution co-manager, academy network managed by National (create + logo + activate), e-certificates (server-side PDF), email notifications (durable queue), national usage-norm + quarterly dashboards with CSV export, public Mentor YUVA Network page, seeding the Director-provided program template as the first draft + the 7 launch academies.

**Explicitly out of scope (v1):**
- WhatsApp notifications (existing stack is not serverless-safe) — email only
- Payments/fees of any kind (concept note: "no financial liability" anywhere)
- MoU document handling (skipped per Director 2026-06-10 — handled offline; no in-app MoU upload/tracking)
- In-app academy-onboarding SoP workflow (the SoP happens offline; national creating the academy IS the approval)
- Zonal/RM layer for the academy app (national + chapter only)
- National↔CII system integrations, public APIs
- Editing the canonical institution master (`yi.institutions`) from this app beyond an "ask to add" request
- Mobile native app (responsive web/PWA only)
- Live video/online session delivery — sessions are physical, at the academy space
- YUVA EC student-committee tooling (EC operates offline; chapter/coordinator accounts are the system surface)
- Private student↔mentor DMs (communication = shared cohort thread in v1)
- Certification-ceremony event tracking (physical events; e-certs support them)

-- Helper: current person (auth.uid() → yi_directory.people.id)
CREATE OR REPLACE FUNCTION yuva.current_person_id() RETURNS uuid
LANGUAGE sql STABLE SECURITY DEFINER SET search_path = yi_directory, public AS $$
  SELECT id FROM yi_directory.people WHERE user_id = auth.uid() AND is_active LIMIT 1;
$$;

-- Role-aware visibility wrapper (shared primitive untouched; tier-model role names kept).
-- The shared primitive yi_directory.current_user_can_see is role-name-AGNOSTIC on its chapter
-- branch (any active app-matching role row with a matching chapter passes), so it MUST NOT be
-- delegated to for the chapter branch — that would grant mentor / institution_coordinator
-- chapter-wide reads. yuva.can_see(p_chapter) returns true ONLY for:
--   (a) the platform tier, via the locked primitive's platform branch;
--   (b) app='yuva' roles yuva_super_admin / yuva_admin (global);
--   (c) app='yuva' role chapter_admin whose yi_chapter matches p_chapter — the chapter branch
--       is implemented natively here with this explicit role whitelist.
-- It must NOT pass mentor or institution_coordinator — those get narrower clauses below.
CREATE OR REPLACE FUNCTION yuva.can_see(p_chapter text) RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER SET search_path = yi_directory, yuva, public AS $$
  SELECT yi_directory.current_user_can_see('yuva', NULL, NULL)   -- platform branch only
  OR EXISTS (
    SELECT 1 FROM yi_directory.people pe
    JOIN yi_directory.role_assignments ra ON ra.person_id = pe.id
    WHERE pe.user_id = auth.uid() AND ra.is_active AND ra.app = 'yuva'
      AND (ra.role IN ('yuva_super_admin','yuva_admin')
           OR (ra.role = 'chapter_admin' AND ra.yi_chapter = p_chapter))
  );
$$;
CREATE OR REPLACE FUNCTION yuva.is_yuva_national() RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER SET search_path = yi_directory, public AS $$
  SELECT EXISTS (
    SELECT 1 FROM yi_directory.people pe
    JOIN yi_directory.role_assignments ra ON ra.person_id = pe.id
    WHERE pe.user_id = auth.uid() AND ra.is_active
      AND ((ra.app = 'yuva' AND ra.role IN ('yuva_super_admin','yuva_admin'))
           OR ra.role = 'super_admin')
  );
$$;

REVOKE SELECT ON yuva.enrollments FROM authenticated;
GRANT SELECT (id, run_id, chapter, person_id, application_id, status, certificate_id, joined_at)
  ON yuva.enrollments TO authenticated;           -- access_code intentionally omitted

**Student session flow:** apply (anon, no login) → cohort formation generates `access_code` per enrollment — codes generated with `crypto.randomBytes` (CSPRNG), 8 chars from the 32-char alphabet (donor: `lib/yi-future/access-code.ts → generateAccessCode`; the donor's `validateAccessCode` has NO rate limiting — do not clone that part) — + acceptance email → student logs in at `/youth-academy/login` via (a) access code → `loginWithAccessCode` (per-IP AND global rate limiting + lockout backoff; throttle state in a small `yuva.login_attempts` table, pruned by the drain cron) → look up enrollment, mint cookie, or (b) email OTP → `requestOtp` (enqueues ONLY if an active enrollment exists for the email, responding identically either way to prevent enumeration; capped per-email and per-IP with backoff) writes hashed 6-digit code to `yuva.login_otps` + enqueues email, `verifyOtp` checks hash/expiry/attempts (max 5), mints cookie. ### Key Technical Decisions

| Decision | Chosen | Alternative | Why |
|---|---|---|---|
| Student identity | Signed `yuva_session` cookie (payload `{personId, exp}` ONLY, ≤24h; enrollments resolved LIVE from the DB per gated action) + access code / email OTP; no auth.users rows | Spec assumption: provision Supabase auth accounts | Matches the proven Yi-Future delegate precedent already encoded in `lib/supabase/middleware.ts`; avoids auth.users pollution + email-confirm friction; reversible later |
| Institution master | FK to `yi.institutions` + one-time backfill absorbing approved `future.colleges` rows | FK to `future.colleges` directly | `yi.institutions` is the designed canonical master (verified live: 60 rows, `source_future_college_id` backlink, `has_yuva_chapter` flag); FKing to Yi-Future's edition-coupled table would create a second institution authority |
| RLS global tier | `yuva.can_see()` wrapper over the locked primitive | Edit `yi_directory.current_user_can_see` role list | Shared primitive is a locked cross-app decision with a pending role-name follow-up; wrapper keeps tier-model role strings AND leaves the primitive untouched |
| Write path | Service client inside gate-first actions; zero INSERT/UPDATE RLS policies | Authenticated-role RLS write policies | Repo-wide pattern (YIP/Yi-Future); students have no auth.uid() so RLS writes are impossible for them anyway; single enforcement point per mutation |
| Anon access | No anon grants on `yuva` at all; public pages read via RSC + service client | Narrow anon SELECT policies | Strictest day-one siloing; avoids the anon-PII-leak class that needed 3 hardening batches in YIP |
| Coordinator binding | `yuva.academies.coordinator_person_id` + role row app='yuva' role='institution_coordinator' | Custom column on `role_assignments` | `role_assignments` is intentionally generic — no per-app columns; academy table is the natural binding |
